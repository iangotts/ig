Executive summary

The Flow PB_Technical_Case_Email_Alerts contains multiple instances of hardcoded Salesforce IDs for email templates within its action elements. No critical stability or error-handling risks were detected, but the hardcoded IDs introduce deployment and maintainability risk, especially across multiple sandboxes and production orgs. Overall risk level: moderate.

Flow Warning Deployment Risk

Why it matters: Hardcoded IDs are org-specific and will break deployments or create drift between environments. After a sandbox refresh or a deploy to production, these template IDs will not resolve — causing failed email sends or unexpected behaviour at scale.

Requirement: Enhance PB_Technical_Case_Email_Alerts Flow to send customer email on Case creation

P2 — Medium Medium Impact Medium Risk Flow · Case Management · Email Alerts · Automation

Expand the existing Flow to send an email notification to the customer when a new Case is created, in addition to its current behaviour for status changes. This requires updating the automation logic to distinguish between Case creation and update events, ensuring the correct email template and recipient are used, and maintaining the integrity of existing email alerts. The change should not disrupt current functionality or introduce duplicate notifications.

Reason: Customers currently do not receive an email when a new Case is created. Adding this notification improves customer communication and case transparency.

Solution: Add a decision element that branches on Case creation versus update, add a new email action for Case creation, and configure the correct template and recipient. Validate that existing status-change emails are unaffected.

Priority, business impact and risk values are demo-defaults and should be edited in the UI.

Residual investigation items

  • Order of Execution review: the update changes whether the automation writes the triggering record; review same-transaction write overlap before implementation.
  • External integrations / middleware: ETL, middleware or API clients may depend on literal strings. Review and update mappings.
  • Managed packages / AppExchange apps: installed packages may include hard-coded labels. Review package configs.
  • Analytics / warehouses / exports: downstream analytics may store historical strings. Decide on mapping or backfill strategy.

Story 1 — Update PB_Technical_Case_Email_Alerts Flow to add create-triggered customer email

User StoryupdateMedium Risk

As a Salesforce administrator, I want the Flow to send an email to the customer when a new Case is created, so that customers are promptly notified of new Cases and communication is improved.

Problem: the Flow only sends emails on Case status changes, not on creation, so customers are not notified when a new Case is opened.

Acceptance criteria
  • AC1: Given a new Case is created, when the Flow is triggered, then an email is sent to the Case's contact using the designated template.
  • AC2: Given a Case is updated (not created), when the Flow is triggered, then only the existing status-change email logic executes and no duplicate or unintended emails are sent.
  • AC3: Given the Flow is updated, when tested in a sandbox, then no errors or duplicate emails occur for either creation or update events.
Implementation details
  • Review the current after-save trigger logic for compatibility with new branching.
  • Add a decision element to branch between Case creation and update.
  • Create a new email action for Case creation with the correct template and recipient.
  • Update the start element's filter formula to include creation logic.
  • Validate no same-transaction write overlap is introduced.
  • Test thoroughly in a sandbox before deployment.
